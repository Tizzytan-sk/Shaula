## 0. TL;DR

桌面宠物 v1（P0+P1+P2）已交付，是时候回答一个更重要的问题：

> **「shaula-agent 接下来 3 个月，最该做哪三件事？」**

答案是这三件（按必须的先后顺序）：

| 优先级 | RFC | 工程量 | 解决什么 |
|------|-----|------|--------|
| **P0** | [RFC-1：ChatApp.tsx 拆分](./2026-06-02-rfc-1-chatapp-split.md) | 6-8 人天 | 4673 行巨石组件，是产品天花板 |
| **P1** | [RFC-2：Agent 协作模式 v0](./2026-06-02-rfc-2-agent-collaboration.md) | 5.5 人天 | 用户不敢放手让 agent 跑超过 10 分钟 |
| **P2** | [RFC-3：Session as Knowledge](./2026-06-02-rfc-3-session-as-knowledge.md) | 12 人天 | 多 session 是核心差异化，但 30+ session 后体验崩盘 |
| **P3** | [RFC-6：Multi-Subagent Collaboration](./2026-06-03-rfc-6-multi-subagent-collaboration.md) | 4-6 人天 MVP | 主 agent 自动派发多个 subagent 并行处理复杂任务 |
| **P4** | [RFC-7：Multi-Agent Final Architecture](./2026-06-04-rfc-7-multi-agent-final-architecture.md) | 10-14 人天 MVP+ | 从并行 subagent 升级为可恢复、可重试、可审计、可验证的多 agent 协同系统 |

**总工程量**：约 **23-25 人天**，按 1 人全职可在 **5-6 周** 完成；按 2 人协作可在 **3 周** 完成。

**不做** 这些事的代价：每多写 1000 行 ChatApp.tsx，重构成本翻倍；每多 10 个用户撞过"agent 跑飞了"，留存掉 5%；每多 50 个 session，sidebar 变成"对话坟场"。**这三件事都是雪球，越早做收益越大。**

## 1. 为什么是这三件事 —— 战略判断

### 1.1 产品定位再确认

shaula-agent 不是「又一个 AI 聊天 UI」，它的定位是：

> **本地优先的、多 session 并行的、带桌面陪伴的 AI coding agent**

跟它相似生态的产品做了横向对比：

| 维度 | Cursor | Cline | Claude Code | **shaula-agent** |
|-----|--------|-------|-------------|----------------|
| 部署模式 | IDE 插件 | VS Code 插件 | CLI | **独立 web/桌面** |
| 多 session | ❌ | ❌ | ❌ | ✅ **核心特性** |
| 桌面陪伴 | ❌ | ❌ | ❌ | ✅ 宠物 v1 |
| Agent 协作 | ✅ Composer | ✅ Plan/Act | ⚠️ approval | ❌ **缺口** |
| Session 检索 | ❌ | ❌ | ❌ | ❌ |
| 项目记忆 | ✅ .cursorrules | ✅ .clinerules | ✅ CLAUDE.md | ❌ |
| Local-first | ⚠️ | ✅ | ✅ | ✅ |

读这张表，**真正能差异化的护城河是**：
1. ✅ **多 session 并行**（结构性差异，难抄）
2. ✅ **桌面陪伴**（已交付，但易抄）
3. 🚧 **agent 协作模式**（行业普遍在卷）
4. 🚧 **session 知识化**（普遍缺口，机会）

**结论**：
- 宠物 v2 / 多端 / 更多模型适配 → **不该做**，因为这些不构成护城河
- 把多 session 这个核心特性**做透**（让它从 5 → 500 还能用） → **必须做**（RFC-3）
- 让 agent 协作能力**追上行业** → **必须做**（RFC-2）
- 而以上两件事都被 ChatApp.tsx 4673 行**卡住** → **先做 RFC-1**

### 1.2 三件事的递进关系

```
                   ┌─────────────────────────────────┐
                   │ RFC-3：Session as Knowledge      │
                   │  让多 session 这个特性能 scale     │
                   └─────────────────────────────────┘
                                  ▲
                                  │
                   ┌─────────────────────────────────┐
                   │ RFC-2：Agent 协作模式 v0          │
                   │  让用户敢用 agent 跑长任务         │
                   └─────────────────────────────────┘
                                  ▲
                                  │
                   ┌─────────────────────────────────┐
                   │ RFC-1：ChatApp.tsx 拆分           │
                   │  让 shaula-agent 还能继续演进       │
                   └─────────────────────────────────┘
```

**RFC-1 是 RFC-2/3 的物理前置条件。** 不先做 RFC-1，RFC-2/3 的所有改动都要塞进 4673 行的 ChatApp.tsx：审批气泡塞一段、搜索框塞一段、摘要逻辑塞一段……三个月后这个文件 7000 行，重构成本翻倍，团队再没人敢碰。

**RFC-2 与 RFC-3 之间无强依赖**，可以并行。但建议 RFC-2 略先一步，因为：
- RFC-2 解决"敢不敢用"，是 retention 第一道闸
- RFC-3 解决"用多了之后体验好不好"，是 retention 第二道闸
- 第一道没过，第二道做了也白做

### 1.3 这三件事**不做**会怎样

| 不做 | 6 个月后 |
|-----|---------|
| RFC-1 | ChatApp.tsx 6000+ 行，每个 PR 都互相 conflict，新人 onboarding 一周读不完 |
| RFC-2 | 用户继续把 agent 当"短任务工具"，长任务全靠 Cursor，我们沦为"小聊天 + 多 session" |
| RFC-3 | 用户 session > 30 后开始放弃 sidebar，每次开 new session，多 session 这个核心特性失去意义 |

→ **三件事的共同特点**：现在不做不会立刻死，但**会逐月削弱产品的根基**，等到不可挽救时已经晚了。

### 1.4 这三件事**做了**会怎样

| 做了 | 6 个月后 |
|-----|---------|
| RFC-1 | 每个新特性都是"在干净接缝上加一段代码"，开发速度 +50%，回归 bug -70% |
| RFC-2 | 用户敢让 agent 跑 30 分钟 deep refactor，长任务份额 +50%，付费意愿 +30% |
| RFC-3 | 老用户 30+ session 不再难用；新用户 onboarding 看到「项目记忆」立刻 wow |

## 2. 推荐执行顺序

按 1 人全职推进的最稳路径（保守估计）：

```
W1   ████░░░░░░░░░░░░  RFC-1 阶段 A：纯抽离（lib/）
W2   ░░██████░░░░░░░░  RFC-1 阶段 B：hook 化（hooks/）
W3   ░░░░░░████░░░░░░  RFC-1 阶段 C：UI 拆分（components/）
W4   ░░░░░░░░██░░░░░░  RFC-2 Phase A：Budget MVP
W5   ░░░░░░░░░░████░░  RFC-2 Phase B：工具审批
W6   ░░░░░░░░░░░░██░░  RFC-3 Phase A：元数据持久化 + lastSeen bug
W7   ░░░░░░░░░░░░░░██  RFC-3 Phase B：搜索 + 摘要
W8   ░░░░░░░░░░░░░░░█  RFC-3 Phase C：项目记忆（v0 改用 SDK AGENTS.md，0 工程）
```

**关键里程碑**：

- **End of W3**：ChatApp.tsx < 800 行，新特性可独立 PR（**对外不变，对内重生**）
- **End of W5**：长任务用户敢放手；高危操作有审批（**对外有感**）
- **End of W8**：sidebar 可搜索；session 有摘要；项目可有 prompt 库（**对外大爆**）

**如果只有 2 周时间能做什么**：
- 必做 RFC-1 阶段 A + B（最关键的去耦）
- 加 RFC-2 Phase A（Budget，1.5d）—— 这是**性价比最高**的产品改进

**如果有 2 人协作怎么排**：
- Person 1：RFC-1 全程（6-8d，3 周）
- Person 2：W1-W2 等 RFC-1 阶段 A 结束 → W3 起做 RFC-3 Phase A（依赖 useSessions） → 平行 RFC-2 Phase A
- W4-W5：Person 2 做 RFC-2 Phase B，Person 1 做 RFC-3 Phase B
- W6：俩人合并打磨 + RFC-3 Phase C

## 3. 三份 RFC 共享的设计原则

写这三份 RFC 时，反复用到下面 5 条原则。它们是 shaula-agent 接下来所有决策的**默认值**：

### P1：渐进迁移，不大爆炸

每一步都可以 **独立 commit、独立 PR、独立 revert**。RFC-1 拆分严格按"数据流职责分层"切，每层完成都是稳定中间态；RFC-2/3 的每个特性都能独立上线。

### P2：默认值即真理

新特性默认开还是默认关，等于产品的隐式 stance。我们的默认值：
- **Budget**：默认开（保护用户钱包）
- **Approval rules**：默认开 1 条（dangerous-bash-destructive，最严的）
- **自动摘要**：默认开（但有日成本上限）
- **项目记忆**：默认 opt-in（避免无脑创建 `.shaula/`）

### P3：本地优先，不偷数据

任何 session 内容**不离开用户机器**（除了 LLM provider）。摘要存本地、索引存本地、记忆存本地。

### P4：诚实标注能与不能

每份 RFC 都明确列了「v0 做」「v0 不做」「需 SDK 配合」。不画大饼。这是给自己的纪律，也是给团队的契约。

### P5：基于真实代码盘点，不凭空设计

写每份 RFC 前都做了 grep / read / 类型签名核对（见各份 RFC 的附录 B）。**最大的发现**：RFC-2 写到一半发现 SDK 已经支持 tool_call 阻断 —— 上一轮盘点是错的，及时更正了方向，把"等 SDK 升级"变成"一次架构投入解锁全部能力"。

## 4. 共享的物理基础设施

三份 RFC 都会用到下面这些**新增的目录/约定**，提前对齐：

```
~/.shaula/                    ← shaula-agent 的全局家目录（RFC-3 引入，RFC-2 复用）
  ├── sessions/{id}.meta.json  ← session 元数据
  ├── search-index.json        ← 全文索引
  └── settings.json            ← 全局设置（budget / rules / summary model）

<project>/.shaula/            ← 项目级目录（RFC-3 引入）
  ├── prompts/*.md
  ├── memory.md
  └── settings.json            ← 项目级覆盖
```

**约定**：`~/.pi/` 是 SDK 的，**我们不动**；`~/.shaula/` 是 shaula-agent 的，自由演进。

## 5. 各 RFC 速览

### 5.1 RFC-1：ChatApp.tsx 拆分

**问题**：4673 行单文件，13 类 state、60+ callback、27 useEffect，handleAgentEvent 单 switch 112 行。

**方案**：拆为 8 个模块（lib/） + 5 个 hooks + 4 个 UI 组件，严格按数据流职责分层。

**收益**：ChatApp.tsx 从 4673 → < 800 行；新特性 incremental 可上。

**风险**：拆错耦合点 → 用"阶段 A 不动行为，只搬代码"打底，单 commit 可 revert。

→ 详见 [RFC-1](./2026-06-02-rfc-1-chatapp-split.md)

### 5.2 RFC-2：Agent 协作模式 v0

**问题**：用户唯一中断手段是 abort，不可恢复 / 没有软上限 / 危险操作来不及拦。

**方案**：3 个特性
1. **会话级 Budget**（cost / turns / duration 上限） — 1.5d
2. **工具审批与改写**（基于 SDK 已有的 tool_call event） — 4d
3. ~~意图预览~~ —— v0 不做

**关键发现**：F2 不需要等 SDK 升级；改造方式是**注入 inline extension**。

**收益**：长任务用户敢放手；高危操作前可干预；成本可预期。

→ 详见 [RFC-2](./2026-06-02-rfc-2-agent-collaboration.md)

### 5.3 RFC-3：Session as Knowledge

**问题**：session 是孤岛 —— sidebar 只能按时间排，没搜索、没摘要、没标签、没项目记忆。`lastSeenMap` 还是内存态（已知 bug）。

**方案**：4 个特性
1. **Session 元数据 + 持久化**（顺手修 lastSeen bug） — 2d
2. **全文检索**（用 minisearch） — 2.5d
3. **自动摘要 + 智能标题**（用 Claude 3.5 Haiku） — 2.5d
4. **项目级记忆 `.shaula/`** — 5d

**收益**：30+ session 不崩盘；session 变知识资产；跨 session 复用 prompt。

**关键风险**：摘要会有 LLM 成本 —— 默认日上限 $0.5。

→ 详见 [RFC-3](./2026-06-02-rfc-3-session-as-knowledge.md)

## 7. 不做的事 —— 显式列表

为避免范围蔓延，明确这些事**在这 3 个月内不做**：

| 不做的事 | 为什么不做 |
|---------|----------|
| 桌面宠物 v2（情绪 / 互动 / 进化） | 不构成护城河，易被抄；不解决核心痛点 |
| 移动端 / iPad 端 | 用户场景不匹配（coding agent 是桌面活儿） |
| 团队协作 / 多人 session | 违反"本地优先"原则；需要后端 |
| 更多模型适配 | SDK 已支持主流；适配新 model 是 SDK 团队的事 |
| Agent 主动求助 / 多 agent 角色 | 等 v0 数据看清形态再说 |
| 全 UI 视觉改版 | 不影响核心价值；本轮专注架构和能力 |
| AI 自动 commit / 自动 PR | Cursor/Cline 已经卷得很猛；我们的差异化不在这 |
| 跨 session 语义检索（embedding） | v0 keyword 搜索够用，验证后再决定 |

## 8. 这套 RFC 之后的下一步

### 8.1 6 个月路线图（猜测，仅供方向参考）

```
M1-M2: RFC-1 + RFC-2          ← 当前 RFC 覆盖
M3:    RFC-3                  ← 当前 RFC 覆盖
─────────────── （以上是本 RFC 系列）─────────────────
M4:    Agent 协作 v1：
         - 跨 session 知识共享（基于 RFC-3 数据）
         - 多 agent 角色（reviewer / coder / qa）
M5:    数据驱动的智能：
         - 自动建议 approval rule
         - 自动建议 prompt 模板
         - "上次类似任务你这样做"
M6:    桌面化深化：
         - Electron menubar / dock 集成
         - 系统级 hotkey
         - 多窗口 session 物理分离
```

### 8.2 数据闸门：什么时候不该继续这条路

如果上线 4 周后观察到：
- 用户使用频次没涨 → 重新评估定位
- 长任务份额没涨 → RFC-2 设计有问题
- session 数量爆涨但摘要被关闭 > 50% → RFC-3 F3 形态错
- 团队没人愿意 review RFC-2 的 extension 改造 → 简化方案 / 推迟

**这些数据是"是否继续按这套 RFC 走"的硬性闸门。**

## 9. 写在最后：审查时请重点关注的三个判断

如果时间紧只能审三件事，请审：

### 判断 1：RFC-1 的拆分粒度是否合理

8 模块 + 5 hooks + 4 组件 = 17 个新文件。会不会过度抽象？vs 4673 行单文件，怎么选？

我的判断：**这个粒度恰好对齐数据流的天然接缝**。再粗会保留耦合，再细会变形式主义。但欢迎反驳。

### 判断 2：RFC-2 的 inline extension 改造是否值得

这是个**架构改造**，比"加个特性"风险高。但它解锁的能力（tool_call 拦截 + 未来 v1+ 的所有 agent 协作）值不值得这次投入？

我的判断：**绝对值得**。SDK 提供这套机制就是给我们用的，不用等于把产品天花板留低。一次性的成本，换永久的能力上限。

### 判断 3：RFC-3 的工程量（12 人天）是不是太大

会不会切成 RFC-3a (F1+F2，5d) + RFC-3b (F3+F4，7d) 更好？

我的判断：**就保持一个 RFC 但分 Phase 上**。这四个特性内部紧密耦合（搜索需要 meta、摘要需要触发器、模板需要 cwd 概念），分开写两份会重复一半内容。
