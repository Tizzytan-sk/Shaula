### 2.3 执行经验沉淀

每个子阶段的稳定打法（之后做类似抽离可直接复用）：

```
1. 在源文件 grep 出目标 JSX 区块 + 它引用的所有 state/callback/prop
2. 列出 props 清单（含类型签名，必须读源码确认，不凭印象）
3. 新建 app/components/Xxx.tsx，1:1 复制 JSX（缩进同步调整）
4. ChatApp 替换为 <Xxx ... /> 单行
5. tsc 校验 → 若 unused imports 报错，grep 校验后批量删
6. 跑 lint（必须持平 91 warning）
7. 跑 build
8. 跑 e2e（必须 6/6）
9. 写 commit message（heredoc，含中文用 -F）→ 提交
```

**踩过 2 次的坑**：

1. **删 unused import 时 grep 假阳性**：被 grep 到的名字可能只是出现在 JSX 注释或字符串字面量里，删了导致 tsc 挂。修正：grep 完二次目视确认调用点。
2. **类型签名凭印象写**：比如 `FilesLayout`、`ReducerState`、`AgentPhase`、`SseStatus`、`StatsSnapshot` 这些自定义类型，必须 read 源文件确认 union 范围 / 可选字段，否则 tsc 编译挂或运行时坐标错位。

### 2.4 < 1200 行目标为什么没达成

RFC-1.5 初始目标是把 ChatApp 切到 < 1200 行。实际只切到 1445 行，差 245 行。原因：

**剩下的不是"视图"，是"函数体"**。ChatApp 现在剩下的内容：

- `useState` × 10+（顶层 UI 状态：rightPanel, sidebarOpen, theme 等）—— **不能抽**，多个组件都要读
- `useEffect` × 8+（订阅 IPC / 监听 keyboard / 同步 pet 等）—— 可抽但需要新建 hook
- 顶层闭包 handler × 20+（onSend / onFork / onApprove 等）—— 可抽但要解决"hook 间共享 state"
- 顶层 useMemo × 5+（派生 state）

要再砍 245 行，必须做的事是 **hook 化函数体**（把闭包 handler 收成 hook 返回的 `{ on*, ... }`），这件事的特征：

- 风险高于"抽视图组件"：handler 之间互相调用，hook 间需要 callback 互传
- 单独成立可写一份 RFC（暂称 RFC-1.6）
- 不在本轮 scope

**判断**：抽完 6 个视图组件后，ChatApp 已经满足"开一个新 modal 不用翻 1889 行代码"的体验目标。剩下 245 行不是"是否可读"问题，是"hook 边界设计"问题，留给独立 RFC。

## 3. RFC-test-infra：测试基建从 0 到 1

### 3.1 背景

RFC-1 在"非目标"里明确写了**不引入新依赖**，所以测试一直没动。但抽完之后 lib/ 里已经有一批纯函数（format / image-utils / chat-reducer），它们：

- 0 DOM / 0 IPC / 0 SSE 依赖
- 是 ChatApp 行为正确性的根基（chat-reducer 错一行整个对话历史就糊）
- 边界条件多（时间格式 / base64 padding / tool_use 回填逻辑）

→ **典型的"该测但没测"代码**。本轮顺手补上测试基建 + 3 个示范，证明这套基建跑得通，后续模块照着加即可。

### 3.2 工具链选择

**为什么用 vitest 不用 jest**：

| 维度 | vitest | jest |
|---|---|---|
| Next 15 + ESM | 原生支持 | 需要 babel-jest 或 swc-jest 配 |
| TypeScript | 原生 | 需 ts-jest |
| 配置文件 | 1 个 vitest.config.ts，复用 vite 生态 | jest.config + babel.config + tsconfig 一坨 |
| 速度（这次 37 cases）| **725ms 全跑** | 通常 2-5s |
| 与 playwright 共存 | `exclude: ['e2e/**']` 一行搞定 | 同 |

vitest 在 Next 项目里的成熟度已经追上来，且这次只测纯 lib 函数，不需要 jsdom，没有任何"踩 jest 老坑"的必要。

### 3.3 配置

`vitest.config.ts`（21 行）：

```ts
import { defineConfig } from "vitest/config";
import path from "node:path";

export default defineConfig({
  test: {
    environment: "node",                          // 纯 lib 函数，不需要 jsdom
    include: ["lib/**/*.test.ts"],                // 限定范围
    exclude: ["node_modules/**", "e2e/**", ".next/**"],
  },
  resolve: {
    alias: {
      "@": path.resolve(__dirname, "."),          // 与 tsconfig @/* 对齐
    },
  },
});
```

`package.json` 新增：

```json
"scripts": {
  "test": "vitest run",        // CI / commit 前用
  "test:watch": "vitest"       // 本地开发用
}
```

### 3.4 测试覆盖

**3 个文件，37 个 test case，0.7 秒跑完**。

#### `lib/format.test.ts`（16 cases）

| 函数 | 覆盖的边界 |
|---|---|
| `formatTokens` | `< 1k`（raw）/ `< 10k`（x.xk）/ `< 1M`（xk）/ `>= 1M`（x.xM） |
| `formatMessageTime` | 空/`!isFinite`/同天 `HH:MM`/跨天 `M月D日 HH:MM` |
| `formatRelativeTime` | 非有限值/`just now`/`Nm ago`/`Nh ago`/`Nd ago`/`>= 7d` 用 locale |
| `shortCwd` | 空/`/Users/x` → `~`/多段 → `…/last2/last1` |

**踩坑**：`formatRelativeTime` 的 `>= 7d` 分支会调 `toLocaleDateString("zh-CN")`，输出受 node 时区/locale 影响。处理方式：测试用 `not.toMatch(/ago$/)` 而非精确字符串匹配，规避环境敏感。

#### `lib/image-utils.test.ts`（8 cases）

| 函数 | 覆盖的边界 |
|---|---|
| `approxBase64Bytes` | 无 padding / 一个 `=` / 两个 `==` / 空串 |
| `formatBytes` | 0 / `<1KB` / `<1MB` / `>= 1MB` |

#### `lib/chat-reducer.test.ts`（13 cases）

最有信息量的一组。`ctxToMessages` 是 session context → ChatMessage[] 的核心转换，错一行整个会话历史就显示错。覆盖：

- 空数组
- user text-only / user text+image（顺序保留）
- assistant text+thinking+image（全 part 映射）
- **tool_use + 后续 tool_result → 回填 result/isError/status=done**（核心 case）
- `is_error=true → status=error`
- 没有 tool_result → `status=running`
- **孤立 `role=tool` message 不出现在输出**（已被合并）
- 跳过未知 role
- 多轮顺序：user → assistant(tool) → tool → user
- `tool_use` 缺 id/name 时跳过该 part

`createInitialState` 顺手测了默认值和 messages 透传。

## 5. 接下来：可选的"RFC-1.6"

如果未来要把 ChatApp.tsx 推到 < 1000 行，下一步是 **hook 化函数体**：

### 5.1 候选清单

ChatApp 剩下的可抽内容：

| 候选 | 估行 | 难度 | 备注 |
|---|---|---|---|
| `useChatHandlers`（onSend / onFork / onRetry 一组）| 200+ | 中 | handler 间互相调用，需共享 ref |
| `useUIShortcuts`（keyboard 监听）| 50+ | 低 | 纯 effect |
| `usePetBridge`（IPC 推送 pet）| 80+ | 中 | 跟 useSessions 有边界耦合 |
| `useAutocomplete` | 100+ | 中 | 已有 state 集中，抽起来顺 |
| `useFiles`（文件拖入 / 粘贴）| 60+ | 低 | 跟 DropOverlay 配对 |

预估全部做完 ChatApp 可降到 **800-900 行**，符合 RFC-1 最初的 `< 800 行` 目标。

### 5.2 为什么先不做

- 风险显著高于"抽视图"：闭包 handler 互相依赖，hook 边界设计错就要回滚
- ChatApp 1445 行已不构成"看不下去"的体验问题
- 测试基建已就位，做 RFC-1.6 时可以"边抽 hook 边补单测"（这才是测试基建的真正价值）
- 应该先看 **RFC-2 / RFC-3 是否会撞 ChatApp**：如果撞，再做 RFC-1.6 顺手处理；如果不撞，可能永远不需要做

### 5.3 测试基建的下一步

37 cases / 3 文件只是起点。优先候选：

| 模块 | 价值 | 难度 |
|---|---|---|
| `lib/chat-reducer.applyEvent` | **最高**（流式事件累积是 chat 行为的核心） | 中（事件序列长，要造 fixture） |

`applyEvent` 单测应该是下一个 RFC（暂称 RFC-test-2）的首选。
