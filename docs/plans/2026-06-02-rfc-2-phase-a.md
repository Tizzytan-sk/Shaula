## 1. 背景：为什么是 Budget 而不是 Approval

### 1.1 RFC-2 的两个候选起点

RFC-2 文档原本规划了 4 个独立能力：

1. **Session Budget**（成本 / 轮次 / 时长上限）
2. **Approval**（敏感工具调用前人工 confirm）
3. **多 agent 协作面板**
4. **共享 context store**

#### 为什么 Phase A 选 Budget：

| 维度 | Budget | Approval | 多 agent | 共享 context |
|---|---|---|---|---|
| 用户痛点紧迫性 | **高**（放手跑会担心烧钱） | 中（已有手动停止） | 低（单 session 已能用） | 低 |
| 实现复杂度 | **低**（纯前端累积 + 阈值判断） | 中（要给每个 tool 加 schema） | 高（多 session UI 大改） | 高（持久化 + 同步） |
| 对 ChatApp 侵入 | **小**（4 处 wire） | 中（要在 tool 流中切入） | 大 | 大 |
| 独立可发布 | **是** | 是 | 否（需 Budget 先做） | 否 |

→ **Budget 投入产出比最高，且不依赖任何其他能力**，是 RFC-2 P1 的天然首选。

### 1.2 用户决策

经过方案讨论，用户拍板：

- 按 Phase A 完整 4 子阶段推进（types → indicator → enforce → settings）
- 不切碎成多个 PR，一次性把 MVP 做透
- 单 agent 单 session 范围内闭环，不跨 agent 共享

执行约定（继承 RFC-1.5 形成的工作流）：

- 不再每步等确认，连续推进
- 每个阶段必须 `tsc + lint(91 warn 持平) + vitest + build + e2e 6/6` 全绿才 commit
- commit message 含中文 → `git commit -F /tmp/commit-msg.txt`
- 抽出/新增代码必须读源码确认类型签名，不凭印象

### 2.2 阶段 A1：数据层 + 持久化 + 单测

**目标**：把 Budget 的"数据"维度抽干净，让上层只关心读写 API。

**产物**：

- `lib/budget/types.ts`（90 行）：`SessionBudget` / `BudgetSpent` / `BudgetStatus` / `BudgetDimension` + `DEFAULT_BUDGET` + localStorage key 常量
- `lib/budget/index.ts`（181 行）：
  - `evaluateBudget(budget, spent)` → `BudgetStatus`（核心纯函数）
  - `normalizeBudget(raw)`：对 undefined / 0 / 负数 / NaN 统一收口为"未启用该维度"
  - `loadGlobalBudget` / `saveGlobalBudget`
  - `loadSessionOverride` / `saveSessionOverride` / `clearSessionOverride`
  - `resolveBudget(agentId)`：override > global > DEFAULT_BUDGET
- `lib/budget/index.test.ts`（334 行，28 cases）：edge case 全覆盖

**关键设计决定**：

| 决策 | 理由 |
|---|---|
| 字段语义：undefined/0/负数/NaN 都视为"未启用" | 让 UI 表单"关闭开关 + 留旧值在 input"零成本可逆 |
| 默认 `{cost:5, turns:30, dur:600s, action:'pause'}` | 5 USD 是用户单次"敢放手"的下限，10min duration 对应 30 轮慢跑 |
| 持久化 key 全 SSR-safe（typeof window 守护） | 避免 Next 15 SSR / app router 误用 |

**门禁**：tsc ✅ / lint 91w 20e ✅ / vitest **65/65（+28）** ✅ / build ✅ / e2e 6/6 ✅

#### 单测踩坑

vitest config 是 `environment: node`，单测里 `localStorage` 不存在。第一次写直接 `localStorage.setItem(...)` 跑挂。

**解法**：用 `vi.stubGlobal('window', { localStorage: memoryStorage })` 注入内存版 storage，然后所有访问都改成 `window.localStorage.xxx`（与生产代码一致）。Setup 一次，28 cases 全跑通。

### 2.3 阶段 A2：useBudget hook + 顶栏 Indicator

**目标**：把 Budget 状态"算出来"并"显示出来"，但不做任何 enforce。

**关键设计**：

#### `runStartedAt` 注入点

duration 维度需要"agent 实际跑了多久"。最干净的注入点是 `useAgentEvents.ts` 的 agent 事件处理：

- `agent_start`：写 `Date.now()` 到 `chatState.runner.runStartedAt`
- `agent_end`：清回 `null`

`lib/session-runner.ts` 同步加字段 + `emptyRunner` 初始化为 null。

#### `useBudget` hook 边界

输入：`activeSnapshot`（chatState）+ `agentId`
输出：`{ budget, spent, status, hasOverride, setSessionOverride, clearOverride }`

duration 只在 `chatState.runner.status === 'streaming'` 时订阅 1s tick，避免 idle 期 setInterval 浪费。

#### 挂载顺序

`useBudget` 必须挂在 `ChatApp.tsx` 第 506 行 `activeSnapshot/agentId` 解构**之后**，否则会读到 stale closure。

#### `BudgetIndicator` 形态

顶栏紧凑形态：⏱ + 进度条（max% 跨三维度）+ 百分比文字。hover tooltip 展开三维度详情（cost USD / turns count / duration mm:ss）。triggered 状态变红。

**门禁**：tsc ✅ / lint 91w 20e ✅ / vitest 65/65 ✅ / build ✅ / e2e 6/6 ✅

### 2.4 阶段 A3：enforcer + 暂停 Modal

**目标**：命中阈值自动调 onAbort 停 turn；pause 模式额外弹窗让用户选择关闭 / 提高上限。

**关键设计**：

#### useBudgetEnforcer 触发去重

最大的坑：status.triggered 在 streaming 中可能多次返回 true（每秒重算），不能每次都 abort。

**解法**：用 `useRef<{ agentId: string; runStartedAt: number } | null>` 记录"本次 run 是否已触发过"。只在 `(agentId, runStartedAt)` 组合首次跃迁到 triggered 时调 onAbort。新 run（runStartedAt 变化）才重置。

#### pause 模式的 onAbort + onPause 双调用

- 先 `onAbort()` 把流式停掉（防止继续烧钱）
- 再 `onPause()` 让父级把 `budgetPausedTrigger` state 设为 truthy，渲染 Modal

stop 模式仅 onAbort，没有 Modal。

#### Modal 的"提高上限并继续"

`handleRaiseAndContinue`：把当前生效 budget 的各启用维度 × 2 写入 session override（不动 global 默认）。这样下一次同 session 继续跑会用 2 倍上限。

**没做**：不自动续发上一条 user message。理由是用户可能改了主意想换 prompt，自动续发反而干扰。用户手动点 Composer 的"继续/重发"按钮即可。

#### 中文引号绕坑

Modal 正文一开始写 `"暂停（命中 budget）"`，ESLint `react/no-unescaped-entities` 直接 error。换成中文 `「」` 引号绕过。

**门禁**：tsc ✅ / lint 91w 20e ✅ / vitest 65/65 ✅ / build ✅ / e2e 6/6 ✅

### 2.5 阶段 A4：Settings 全局默认区块

**目标**：用户能在 Settings 改 Budget 默认值，自动持久化到 localStorage（pi-budget key）。

**关键设计**：

#### 两个 Panel 都要嵌

`SettingsPanel.tsx` 内部有 Electron 和 Web 两个独立 panel function：

- `WebSettingsPanel`（Web 模式）main 第 134 行附近
- 默认 export `SettingsPanel`（Electron 模式）main 第 452 行附近

→ 同一个 `<BudgetSettingsSection />` 在两处各挂一份，不抽 layout（侵入更深、收益更低）。

#### Hydration mismatch 的两次方案演进

| 方案 | 问题 |
|---|---|
| v1：useState 初值用 `loadGlobalBudget()` | SSR 期间 `window` undefined，loadGlobalBudget 返回 DEFAULT → CSR mount 后 localStorage 有值 → React 报 hydration mismatch |
| v2：useState 初值用 DEFAULT + useEffect mount 后 setX | 解决 hydration 但 4 个 setX in effect → lint **+1 warning（92）** 违反 91 持平偏好 |
| v3（最终）：拆成 wrapper + inner，wrapper 用 `next/dynamic({ ssr: false })`，inner 用 useState lazy init 直接读 localStorage | 干净 |

v3 的代价：多一个文件（22 行 wrapper）。收益：**lint 91 持平 + 单一信源 + 0 useEffect**。值得。

#### "无 Save 按钮"决策

每次 change → 立刻 `saveGlobalBudget`。理由是用户体验：Budget 是 4 个滑杆级别的简单配置，加 Save 按钮反而增加遗忘风险。所有写 op 都经 normalizeBudget 收口，0 校验风险。

**门禁**：tsc ✅ / lint **91w** 20e ✅ / vitest 65/65 ✅ / build ✅ / e2e 6/6 ✅

## 3. 数据流全景

```
┌─────────────────┐                           ┌──────────────────┐
│ Settings Panel  │ saveGlobalBudget          │ Budget Modal     │
│ (pi-budget)     │ ◄────────────────┐        │ "提高上限并继续" │
└─────────────────┘                  │        └────────┬─────────┘
       │ load                        │                 │ saveSessionOverride
       ▼                             │                 │ (pi-budget-override-<id>)
  ┌────────────────────────────────────────────────────▼──────┐
  │ resolveBudget(agentId) = override > global > DEFAULT       │
  └────────────────────────────────────────────────────────────┘
                          │
                          ▼
              ┌──────────────────────┐
              │  useBudget hook      │
              │  budget + spent →    │
              │  evaluateBudget →    │
              │  status (triggered?) │
              └──────────────────────┘
                  │            │
                  ▼            ▼
       ┌────────────────┐  ┌─────────────────────────┐
       │ BudgetIndicator│  │ useBudgetEnforcer       │
       │ (顶栏 HUD)     │  │ (status 跃迁 → onAbort) │
       └────────────────┘  └─────────┬───────────────┘
                                     │ pause 时
                                     ▼
                          ┌─────────────────────┐
                          │ BudgetExceededModal │
                          └─────────────────────┘
```

三个 spent 维度的数据源：

| 维度 | 数据源 | 计算时机 |
|---|---|---|
| cost (USD) | `chatState.runner.totals.cost_usd` | agent 事件累积 |
| turns | `chatState.messages.filter(m => m.role === 'user').length` | useBudget 每次重算 |
| duration (sec) | `(Date.now() - runner.runStartedAt) / 1000` | streaming 中 1s tick |

## 5. 未做的 / 留给 Phase B

### 5.1 已显式 out-of-scope

| 项 | 留给 | 原因 |
|---|---|---|
| 多 agent 共享 budget 池 | Phase B | 需要先做 RFC-3 的 session 元数据系统 |
| 团队 / 组织级 budget | 后续 RFC | 需要后端账户系统 |
| Budget 命中后自动续发上一条 user message | Phase B 可选 | 当前手动重发即可，自动续发有"撞墙循环"风险 |
| 历史 budget 命中事件审计 | Phase B | 需要 RFC-3 的事件持久化 |
| Per-tool cost cap（某些贵 tool 单独限） | Phase C | 需要 RFC-2 §6 的 Approval 框架 |

### 5.2 已知 sharp edges

1. **cost 数据准确性依赖 `chatState.runner.totals.cost_usd`**：当前是 agent 事件累积，如果上游 model 漏报 usage（某些供应商在 stream 异常断开时），cost 会偏低。Phase A 不解决。
2. **duration 不算"agent 思考时间外"**：用户阅读 / 编辑 prompt 的时间也计入 duration（因为 runStartedAt 只在 agent_start 写）。这是有意：duration 防的是"agent 跑飞"，不是"用户磨蹭"。
3. **session override 不会自动 GC**：localStorage 里会越积越多 `pi-budget-override-<id>`。Phase B 应该加"删除 session 时同步删 override"。

## 6. 复盘：本轮做对的 4 件事

1. **MVP 一锅出**：4 个阶段一次推完，没有"先 demo 一半"。中间态对用户都是半成品体验（有 Indicator 没 enforce 反而误导），分多次发是负优化。
2. **每个阶段独立 commit + 全门禁**：4 个 commit 4 次完整门禁，任何一个出问题都能精确回滚。
3. **A4 hydration 问题敢走第三版**：v2 +1 warning 是技术上能交付的，但违反 91 持平偏好。多写 22 行 wrapper 换 0 漂移，符合"不喜欢零散打补丁"的偏好。
4. **lib/budget 是纯函数 + 28 单测**：核心阈值判断 `evaluateBudget` 0 副作用、edge case 全覆盖，未来改阈值语义只改一处。

## 7. 复盘：本轮可以更好的 2 件事

1. **A4 hydration 一开始就该走 v3**：v1 / v2 浪费了一轮编辑时间。教训：**只要组件依赖 localStorage 且首屏需要展示数据，直接上 `next/dynamic({ ssr: false })`**，不要先试 useEffect 同步。
2. **Modal 的"提高上限并继续"语义可能要回测**：当前是 × 2 写 override 后等用户重发。Phase B 上线时收到的第一个反馈很可能是"为什么没自动继续"，需要文案进一步明确（按钮文字改成"× 2 上限（需手动重发）"？）。
